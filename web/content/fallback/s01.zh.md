# s01: Provider 无关客户端 — 一个接口，任意 Provider

[中文](README.md) · [English](README.en.md)

`s01` → [s02](../s02_middleware_pipeline/) → s03 → s04 → ... → s20
> *"Write once, run against any LLM"* — IChatClient 把 Provider 抽象掉了。你的代码永远不需要关心背后是哪个模型。
>
> **Harness 层**: 基础 — 后续每一章都建立在这个单一接口之上。

---

## 问题

你基于 OpenAI 的 API 构建了一个聊天应用，效果很好。然后团队想在某个功能上试试 Claude。或者为了开发方便切换到本地模型。或者为了节省成本用 DeepSeek。

每个 Provider 都有自己的 SDK、自己的请求格式、自己的流式协议。切换意味着重写集成代码、修改每个调用点、从头测试。

你希望代码只写一次，通过改配置文件就能切换 Provider。

---

## 解决方案

Microsoft.Extensions.AI 引入了 `IChatClient` — 一个所有 Provider 都实现的统一接口。应用代码调用 `IChatClient` 方法，具体的实现在启动时确定，而不是硬编码在代码中。

```csharp
IChatClient chatClient = new ChatClient(modelId,
    new ApiKeyCredential(apiKey),
    new OpenAIClientOptions { Endpoint = new Uri(baseUrl) })
    .AsIChatClient();
```

`.AsIChatClient()` 就是桥梁。它把任何兼容 OpenAI 的客户端包装到同一个 `IChatClient` 接口后面。切换 Provider 只需修改构造方式 — 下游代码完全不变。

---

## 工作原理

**第 1 步**：从配置读取 Provider 连接信息。

```csharp
var config = new ConfigurationBuilder()
    .AddJsonFile("appsettings.json", optional: true)
    .AddEnvironmentVariables()
    .Build();

var baseUrl = config["baseUrl"] ?? "https://api.openai.com/v1";
var apiKey = config["apiKey"] ?? Environment.GetEnvironmentVariable("OPENAI_API_KEY");
var modelId = config["modelId"] ?? "gpt-4o-mini";
```

配置与代码分离。同一个二进制文件可以连接 OpenAI、DeepSeek、Ollama 或任何兼容端点。

**第 2 步**：创建 IChatClient。

```csharp
IChatClient chatClient = new ChatClient(modelId,
    new ApiKeyCredential(apiKey),
    new OpenAIClientOptions { Endpoint = new Uri(baseUrl) })
    .AsIChatClient();
```

`IChatClient` 接口暴露两个核心方法：`GetResponseAsync` 用于非流式调用，`GetStreamingResponseAsync` 用于流式调用。每个 Provider 都实现了这两个方法。

**第 3 步**：调用模型 — 非流式。

```csharp
var response = await chatClient.GetResponseAsync(
    "什么是 .NET？一句话回答。");
Console.WriteLine(response.Text);
```

响应对象提供文本、token 计数、完成原因和模型请求的工具调用。

**第 4 步**：调用模型 — 流式。

```csharp
await foreach (var update in chatClient.GetStreamingResponseAsync(
    "列举 C# 的三个优点。"))
{
    Console.Write(update);
}
```

流式返回在数据到达时逐块输出。每个更新是一个可以立即渲染的部分响应。无论后端是 OpenAI、Anthropic 还是本地模型，接口完全相同。

**第 5 步**：通过修改配置切换 Provider。

```text
# OpenAI（默认）
baseUrl=https://api.openai.com/v1
modelId=gpt-4o-mini

# DeepSeek
baseUrl=https://api.deepseek.com/v1
modelId=deepseek-chat

# 本地 Ollama
baseUrl=http://localhost:11434/v1
modelId=llama3
```

不需要改代码，不需要重新编译。改配置，重启即可。

## 抽象边界

`IChatClient` 不仅仅是一个便利封装。它定义了整个 MEAI 生态系统构建的契约：

| 能力 | 通过 IChatClient 如何工作 |
|------|--------------------------|
| **流式传输** | `GetStreamingResponseAsync()` 返回 `IAsyncEnumerable<ChatResponseUpdate>` |
| **非流式** | `GetResponseAsync()` 返回包含文本、用量、完成原因的 `ChatResponse` |
| **工具调用** | 模型可以请求工具调用；客户端将其作为结构化块暴露 |
| **中间件** | `.AsBuilder().Use(...)` 用可组合层包装客户端（见 s02） |
| **DI 集成** | `AddChatClient()` 在 ASP.NET Core 的服务容器中注册客户端 |

从 s02 开始的每一章都使用 `IChatClient`，不需要知道也不关心背后是哪个 Provider。

## 试一下

**准备**（首次运行）：

```sh
dotnet build
cp s01_provider_agnostic/appsettings.example.json s01_provider_agnostic/appsettings.json
# 编辑 appsettings.json，把 PUT-YOUR-KEY-HERE 替换成你的 API key
# 或设置环境变量 OPENAI_API_KEY=sk-...
```

**运行**：

```sh
dotnet run --project s01_provider_agnostic
```

观察重点：

1. 非流式调用一次性返回完整响应
2. 流式调用在 token 到达时逐字打印
3. 中间件管道用 `[middleware] →` 和 `← [middleware]` 标记包装调用

试试在 `appsettings.json` 中切换 Provider 再运行。代码不需要任何改动。

## 接下来

现在我们有了 Provider 无关的客户端，下一个问题是：如何在不修改客户端本身的情况下添加日志、计时、缓存等横切关注点？

→ s02 中间件管道：`DelegatingChatClient` 用可组合的中间件层包装 `IChatClient`。

<details>
<summary>深入 MEAI 设计</summary>

> `IChatClient` 的设计模式参照了 .NET 中的 `HttpClient`。正如 `HttpClient` 抽象了 HTTP 传输，`IChatClient` 抽象了 LLM 传输。关键设计决策：

<details>
<summary>1. 为什么是接口而不是抽象类？</summary>

接口允许 Provider 在现有类层次结构旁边实现 `IChatClient`。OpenAI SDK 的 `ChatClient` 通过扩展方法获得 `.AsIChatClient()` — 不需要继承。这意味着 MEAI 和任何特定 SDK 之间零耦合。

</details>

<details>
<summary>2. 流式和非流式是独立路径</summary>

`GetResponseAsync` 返回完整的 `ChatResponse`。`GetStreamingResponseAsync` 返回 `IAsyncEnumerable<ChatResponseUpdate>`。它们是独立的方法，而不是布尔标志，因为流式响应有根本不同的生命周期语义 — 你在数据块到达时处理它们，而不是在完整响应缓冲后处理。

</details>

<details>
<summary>3. Provider 切换是部署关注点，不是代码关注点</summary>

推荐模式是在启动时将 `IChatClient` 注册到 DI 容器。应用代码依赖接口，永远不依赖具体的 Provider。这使得：
- 开发时使用快速的本地模型
- 生产时切换到云模型
- 无需代码更改即可 A/B 测试不同 Provider
- 主 Provider 不可用时回退到备用 Provider

</details>

**一句话**：`IChatClient` 就是 LLM 世界的 `HttpClient` — 一个接口，任意 Provider，零锁定。

</details>
