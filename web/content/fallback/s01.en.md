---

## The Problem

You build a chat app against OpenAI's API. It works great. Then your team wants to try Claude for a specific feature. Or switch to a local model for development. Or use DeepSeek for cost savings.

Each provider has its own SDK, its own request format, its own streaming protocol. Switching means rewriting your integration code, updating every call site, and testing everything from scratch.

You want to write the code once and change providers by editing a config file.

---

## The Solution

Microsoft.Extensions.AI introduces `IChatClient` — a single interface that every provider implements. Your application code calls `IChatClient` methods. The concrete implementation behind it is determined at startup, not baked into your code.

```csharp
IChatClient chatClient = new ChatClient(modelId,
    new ApiKeyCredential(apiKey),
    new OpenAIClientOptions { Endpoint = new Uri(baseUrl) })
    .AsIChatClient();
```

That `.AsIChatClient()` call is the bridge. It wraps any OpenAI-compatible client behind the same `IChatClient` interface. To switch providers, you change the construction — everything downstream stays identical.

---

## How It Works

**Step 1**: Configure the provider connection from settings.

```csharp
var config = new ConfigurationBuilder()
    .AddJsonFile("appsettings.json", optional: true)
    .AddEnvironmentVariables()
    .Build();

var baseUrl = config["baseUrl"] ?? "https://api.deepseek.com/v1";
var apiKey = config["apiKey"] ?? Environment.GetEnvironmentVariable("DEEPSEEK_API_KEY");
var modelId = config["modelId"] ?? "deepseek-chat";
```

Configuration is external to code. The same binary can hit OpenAI, DeepSeek, Ollama, or any compatible endpoint.

**Step 2**: Create the IChatClient.

```csharp
IChatClient chatClient = new ChatClient(modelId,
    new ApiKeyCredential(apiKey),
    new OpenAIClientOptions { Endpoint = new Uri(baseUrl) })
    .AsIChatClient();
```

The `IChatClient` interface exposes two core methods: `GetResponseAsync` for non-streaming and `GetStreamingResponseAsync` for streaming. Every provider implements both.

**Step 3**: Call the model — non-streaming.

```csharp
var response = await chatClient.GetResponseAsync(
    "What is .NET? Answer in one sentence.");
Console.WriteLine(response.Text);
```

The response object gives you the text, token counts, finish reason, and any tool calls the model requested.

**Step 4**: Call the model — streaming.

```csharp
await foreach (var update in chatClient.GetStreamingResponseAsync(
    "Name three benefits of C#."))
{
    Console.Write(update);
}
```

Streaming returns chunks as they arrive. Each update is a partial response that you can render immediately. The interface is identical regardless of whether the backend is OpenAI, Anthropic, or a local model.

**Step 5**: Switch providers by changing config.

```text
# DeepSeek (default)
baseUrl=https://api.deepseek.com/v1
modelId=deepseek-chat

# OpenAI
baseUrl=https://api.openai.com/v1
modelId=gpt-4o-mini

# Local Ollama
baseUrl=http://localhost:11434/v1
modelId=llama3
```

No code change. No recompile. Just a config edit and restart.

## The Abstraction Boundary

`IChatClient` is not just a convenience wrapper. It defines a contract that the entire MEAI ecosystem builds on:

| Capability | How it works through IChatClient |
|-----------|--------------------------------|
| **Streaming** | `GetStreamingResponseAsync()` returns `IAsyncEnumerable<ChatResponseUpdate>` |
| **Non-streaming** | `GetResponseAsync()` returns `ChatResponse` with text, usage, finish reason |
| **Tool calls** | Models can request tool invocations; the client surfaces them as structured blocks |
| **Middleware** | `.AsBuilder().Use(...)` wraps the client with composable layers (see s02) |
| **DI integration** | `AddChatClient()` registers the client in ASP.NET Core's service container |

Every chapter from s02 onward uses `IChatClient` without knowing or caring which provider is behind it.

<details>
<summary>Dive into MEAI Design</summary>

> `IChatClient` is modeled after the `HttpClient` pattern in .NET. Just as `HttpClient` abstracts HTTP transport, `IChatClient` abstracts LLM transport. The key design decisions:

<details>
<summary>1. Why an interface, not an abstract class?</summary>

Interfaces allow providers to implement `IChatClient` alongside their existing class hierarchies. The OpenAI SDK's `ChatClient` gets `.AsIChatClient()` as an extension method — no inheritance required. This means zero coupling between MEAI and any specific SDK.

</details>

<details>
<summary>2. Streaming and non-streaming are separate paths</summary>

`GetResponseAsync` returns a complete `ChatResponse`. `GetStreamingResponseAsync` returns `IAsyncEnumerable<ChatResponseUpdate>`. They are separate methods, not a boolean flag, because streaming responses have fundamentally different lifecycle semantics — you process chunks as they arrive, not after the full response is buffered.

</details>

<details>
<summary>3. Provider switching is a deployment concern, not a code concern</summary>

The recommended pattern is to register `IChatClient` in the DI container at startup. Application code depends on the interface, never on a concrete provider. This makes it possible to:
- Use a fast local model for development
- Switch to a cloud model for production
- A/B test different providers without code changes
- Fall back to a secondary provider if the primary is down

</details>

**In one sentence**: `IChatClient` is the `HttpClient` of LLMs — one interface, any provider, zero lock-in.

</details>
