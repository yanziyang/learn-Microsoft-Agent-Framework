>
> **Harness Layer**: Production — running agents as services with distributed tracing.

---

## The Problem

Your agent works locally. You run it with `dotnet run`, it processes a request, and exits. But production agents need to:

- Accept HTTP requests from clients
- Start up and shut down gracefully
- Report their health status
- Emit traces so you can debug what happened during a request
- Be restartable, scalable, and monitorable

Running an agent as a console app is fine for development. Production needs infrastructure.

---

## The Solution

ASP.NET Core's hosting model provides everything an agent needs. `AddAIAgent()` registers the agent as a first-class service. `UseOpenTelemetry()` adds distributed tracing around every LLM call.

```csharp
var builder = WebApplication.CreateBuilder(args);

// Register IChatClient with middleware
builder.Services.AddSingleton<IChatClient>(instrumentedClient);

// Register the agent
builder.AddAIAgent("assistant",
    instructions: "You are a helpful assistant.");

var app = builder.Build();
app.MapPost("/chat", async (string message, IChatClient client) => { ... });
app.Run();
```

The agent participates in the same lifecycle, health check, and configuration infrastructure as any other hosted service.

## How It Works

**Step 1**: Set up OpenTelemetry tracing.

```csharp
var serviceName = "AgentTutorial";
var sourceName = "AgentTracing";
using var tracerProvider = Sdk.CreateTracerProviderBuilder()
    .SetResourceBuilder(ResourceBuilder.CreateDefault()
        .AddService(serviceName))
    .AddSource(sourceName)
    .AddHttpClientInstrumentation()
    .AddOtlpExporter()
    .Build();
```

This creates a trace provider that:
- Names the service "AgentTutorial"
- Listens for activities from "AgentTracing"
- Instruments HTTP client calls automatically
- Exports traces via OTLP (compatible with Jaeger, Zipkin, Application Insights)

**Step 2**: Create and instrument the IChatClient.

```csharp
IChatClient baseClient = new OpenAIClient(
    new ApiKeyCredential(apiKey),
    new OpenAIClientOptions { Endpoint = new Uri(baseUrl) })
    .GetChatClient(modelId).AsIChatClient();

var instrumentedClient = baseClient.AsBuilder()
    .UseFunctionInvocation()
    .UseOpenTelemetry(sourceName: sourceName,
        configure: c => c.EnableSensitiveData = true)
    .Build();
```

The middleware pipeline wraps the base client with:
- `UseFunctionInvocation()` — automatic tool dispatch (see s04)
- `UseOpenTelemetry()` — tracing spans for every LLM call

Every request through `instrumentedClient` emits trace data automatically.

**Step 3**: Register services and the agent.

```csharp
builder.Services.AddSingleton<IChatClient>(instrumentedClient);
builder.AddAIAgent("assistant",
    instructions: "You are a helpful assistant.");
```

`AddAIAgent()` registers the agent in the DI container. It becomes available for dependency injection in controllers, minimal APIs, or other services.

**Step 4**: Define API endpoints.

```csharp
app.MapPost("/chat", async (string message, IChatClient client) =>
{
    using var activity = new ActivitySource(sourceName)
        .StartActivity("ManualChat");
    var response = await client.GetResponseAsync(message);
    return Results.Ok(new { response = response.Text });
});

app.MapGet("/health", () =>
    Results.Ok(new { status = "healthy",
        timestamp = DateTime.UtcNow }));
```

Two endpoints:
- `POST /chat` — accepts a message, runs it through the agent, returns the response
- `GET /health` — returns health status for load balancers and orchestrators

**Step 5**: Run as a hosted service.

```sh
dotnet run --project s19_hosting_observability
```

The agent now:
- Listens for HTTP requests
- Emits OpenTelemetry traces for every LLM call
- Reports health status
- Shuts down gracefully on SIGTERM

## What OpenTelemetry Gives You

Every LLM call through the instrumented client automatically produces trace spans:

| Span attribute | What it captures |
|---------------|-----------------|
| `gen_ai.system` | Provider name (openai, etc.) |
| `gen_ai.request.model` | Model ID requested |
| `gen_ai.response.finish_reason` | Why the model stopped |
| `gen_ai.usage.prompt_tokens` | Input token count |
| `gen_ai.usage.completion_tokens` | Output token count |
| `duration_ms` | End-to-end latency |

These spans flow to any OTLP-compatible backend: Jaeger, Zipkin, Application Insights, Grafana Tempo, or any other observability platform.

## Try It

**Setup** (first run):

```sh
dotnet build
cp s19_hosting_observability/appsettings.example.json s19_hosting_observability/appsettings.json
# Edit appsettings.json with your API key
```

**Run**:

```sh
dotnet run --project s19_hosting_observability
```

Test with curl:

```sh
# Health check
curl http://localhost:5000/health

# Chat
curl -X POST http://localhost:5000/chat \
  -H "Content-Type: application/json" \
  -d '"What is .NET?"'
```

What to observe:

1. The health endpoint returns a JSON status
2. The chat endpoint processes your message through the agent
3. Console output shows OpenTelemetry trace data (if an OTLP exporter is configured)

<details>
<summary>Dive into Hosting Design</summary>

> The hosting model follows ASP.NET Core's standard patterns. The agent is just another service in the DI container.

<details>
<summary>1. Why AddAIAgent() instead of manual registration?</summary>

`AddAIAgent()` handles agent-specific concerns: session management, tool registration, instruction binding, and lifecycle hooks. Manual registration would require you to wire all of these yourself. The extension method encapsulates the complexity behind a single call.

</details>

<details>
<summary>2. OpenTelemetry middleware vs manual instrumentation</summary>

The middleware approach (`UseOpenTelemetry()`) automatically traces every LLM call without modifying application code. Manual instrumentation (`Activity.Start/Stop`) gives more control but requires wrapping every call site. Middleware is the recommended approach for consistency and maintainability.

</details>

<details>
<summary>3. Health checks are free with ASP.NET Core</summary>

The `/health` endpoint is a minimal API route. In production, you'd use ASP.NET Core's health check middleware (`app.MapHealthChecks()`) which integrates with Kubernetes liveness probes, load balancers, and monitoring dashboards. The agent's health depends on the LLM client's availability and any external dependencies.

</details>

**In one sentence**: Hosting turns your agent from a script into a service — observable, scalable, and production-ready.

</details>
