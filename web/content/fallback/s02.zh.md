# s02: 中间件管道 — 像洋葱一样层层叠加

[中文](README.md) · [English](README.en.md)

[s01](../s01_provider_agnostic/) → `s02` → [s03](../s03_agent_loop/) → s04 → ... → s20
> *"Wrap once, inspect everything"* — 每个中间件都能看到所有请求和响应，而不碰核心客户端。
>
> **Harness 层**: 基础 — 围绕聊天客户端的可组合横切关注点。

---

## 问题

你在 s01 有了一个可用的 `IChatClient`。现在你想加日志来看模型收到了什么。然后加计时来测量延迟。然后加缓存来避免重复调用。然后加重试逻辑来处理瞬态故障。

朴素做法：修改调用点。在每次调用前后加 `Console.WriteLine`。用 `Stopwatch` 包装。检查缓存字典。加 try-catch 和退避逻辑。

每个新行为都会散布到代码库中。每个调用点都需要同样的样板代码。测试一个关注点意味着要拆开所有其他的。

---

## 解决方案

`DelegatingChatClient` 是中间件基类。每个中间件包装链中的下一个客户端，在核心周围形成洋葱结构：

```text
请求  → [计时] → [日志] → [OpenAI 客户端]
响应 ← [计时] ← [日志] ← [OpenAI 客户端]
```

每一层处理一个关注点。核心客户端保持不变。添加或删除中间件只需修改一行管道代码。

---

## 工作原理

**第 1 步**：从 s01 的基础客户端开始。

```csharp
IChatClient innerClient = new ChatClient(modelId,
    new ApiKeyCredential(apiKey),
    new OpenAIClientOptions { Endpoint = new Uri(baseUrl) })
    .AsIChatClient();
```

**第 2 步**：构建中间件管道。

```csharp
var client = innerClient
    .AsBuilder()
    .Use(inner => new TimingChatClient(inner))
    .Use(inner => new LoggingChatClient(inner))
    .Build();
```

`.AsBuilder()` 创建一个 `ChatClientBuilder`。每个 `.Use()` 用一个新的中间件层包装当前客户端。最后添加的 `.Use()` 是最外层 — 它最先看到请求，最后看到响应。

**第 3 步**：实现一个中间件 — TimingChatClient。

```csharp
class TimingChatClient(IChatClient innerClient)
    : DelegatingChatClient(innerClient)
{
    public override async Task<ChatResponse> GetResponseAsync(
        IEnumerable<ChatMessage> messages,
        ChatOptions? options = null,
        CancellationToken cancellationToken = default)
    {
        var sw = Stopwatch.StartNew();
        var response = await base.GetResponseAsync(
            messages, options, cancellationToken);
        sw.Stop();
        Console.WriteLine($"  [timing] {sw.ElapsedMilliseconds}ms");
        return response;
    }
}
```

模式始终相同：
1. 在调用 `base.*` **之前**做某事
2. 调用 `base.*` 传递到下一层
3. 在响应返回**之后**做某事
4. 返回响应（或 yield 流式更新）

**第 4 步**：实现一个中间件 — LoggingChatClient。

```csharp
class LoggingChatClient(IChatClient innerClient)
    : DelegatingChatClient(innerClient)
{
    public override async Task<ChatResponse> GetResponseAsync(
        IEnumerable<ChatMessage> messages,
        ChatOptions? options = null,
        CancellationToken cancellationToken = default)
    {
        var lastUserMsg = messages
            .LastOrDefault(m => m.Role == ChatRole.User)?.Text;
        Console.WriteLine($"  [logging] Q: {lastUserMsg?[..50]}...");
        var response = await base.GetResponseAsync(
            messages, options, cancellationToken);
        Console.WriteLine($"  [logging] A: {response.Text?.Length} 字符");
        return response;
    }
}
```

每个中间件是独立的。你可以用 mock 客户端包装来单独测试 `LoggingChatClient`。你可以在任何管道中复用它。

**第 5 步**：调用管道 — 客户端工作方式完全相同。

```csharp
var response = await client.GetResponseAsync(
    "什么是依赖注入？一句话回答。");
```

调用者不知道也不关心中间件的存在。响应是一样的。副作用（日志、计时）透明地发生。

## 管道顺序很重要

```text
.Use(inner => new TimingChatClient(inner))    // 最外层
.Use(inner => new LoggingChatClient(inner))   // 中间层
// OpenAI 客户端                              // 最内层
```

**请求流**: 计时 → 日志 → OpenAI
**响应流**: OpenAI → 日志 → 计时

把横切关注点（日志、跟踪、指标）放在外面。把业务逻辑中间件（重试、缓存、函数调用）放在靠近客户端的位置。

MEAI 提供了内置中间件，可以在管道中任意位置插入：

| 中间件 | 功能 |
|--------|------|
| `.UseFunctionInvocation()` | 自动分发工具调用（见 s04） |
| `.UseOpenTelemetry()` | 添加分布式跟踪 span |
| `.UseDistributedCache()` | 添加响应缓存 |

**运行**：

```sh
dotnet run --project s02_middleware_pipeline
```

观察重点：

1. `[logging]` 中间件打印问题和响应长度
2. `[timing]` 中间件打印耗时毫秒数
3. 顺序：计时包装日志，所以计时包含日志的开销
4. 流式调用展示相同的中间件行为，但是逐块处理

## 接下来

我们有了带可组合中间件的 Provider 无关客户端。但模型只能回答问题 — 它不能在现实世界中采取行动。

→ s03 Agent 循环：`while True` 循环让模型调用工具、执行工具、并基于结果继续推理。

<details>
<summary>深入中间件设计</summary>

> MEAI 中的中间件模式与 ASP.NET Core 中间件使用相同的装饰器链。每个 `DelegatingChatClient` 持有对 `InnerClient` 的引用并委托给它。

<details>
<summary>1. 为什么用 DelegatingChatClient 而不是事件或钩子？</summary>

事件和钩子是解耦的，但难以推理执行顺序。装饰器链是显式的：你可以通过阅读管道构造看到中间件的确切顺序。每一层包装下一层，使测试和组合变得简单。

</details>

<details>
<summary>2. 流式需要单独重写</summary>

`GetResponseAsync` 处理非流式。`GetStreamingResponseAsync` 处理流式。如果你的中间件需要观察所有流量，必须同时重写两个方法。流式重写使用 `IAsyncEnumerable` 和 `yield return` 来透明地传递数据块。

```csharp
public override async IAsyncEnumerable<ChatResponseUpdate>
    GetStreamingResponseAsync(
        IEnumerable<ChatMessage> messages,
        ChatOptions? options = null,
        [EnumeratorCancellation] CancellationToken ct = default)
{
    // 之前
    await foreach (var update in base.GetStreamingResponseAsync(
        messages, options, ct))
    {
        yield return update;  // 传递
    }
    // 之后
}
```

</details>

<details>
<summary>3. FunctionInvokingChatClient 只是中间件</summary>

工具分发不是特殊的。`FunctionInvokingChatClient` 是一个 `DelegatingChatClient`，它拦截 `tool_use` 块、执行工具并将结果反馈到对话中。它与日志和缓存的组合方式与其他中间件完全相同。这是 s04 的主题。

</details>

**一句话**：中间件就是应用于 LLM 调用的洋葱模式 — 每一层添加行为而不改变核心。

</details>
