# s02: Middleware Pipeline — Stack Behavior Like Onion Layers

[中文](README.md) · [English](README.en.md)

[s01](../s01_provider_agnostic/) → `s02` → [s03](../s03_agent_loop/) → s04 → ... → s20
> *"Wrap once, inspect everything"* — Each middleware sees every request and response without touching the core client.
>
> **Harness Layer**: Foundation — composable cross-cutting concerns around the chat client.

---

## The Problem

You have a working `IChatClient` from s01. Now you want to add logging to see what the model receives. Then timing to measure latency. Then caching to avoid redundant calls. Then retry logic for transient failures.

The naive approach: modify the call site. Add `Console.WriteLine` before and after each call. Wrap with a `Stopwatch`. Check a cache dictionary. Add a try-catch with backoff.

Every new behavior spreads across your codebase. Each call site needs the same boilerplate. Testing one concern means unwrapping all the others.

---

## The Solution

`DelegatingChatClient` is the middleware base class. Each middleware wraps the next client in the chain, forming an onion around the core:

```text
Request  → [Timing] → [Logging] → [OpenAI Client]
Response ← [Timing] ← [Logging] ← [OpenAI Client]
```

Each layer handles one concern. The core client stays unchanged. Adding or removing middleware is a one-line pipeline change.

---

## How It Works

**Step 1**: Start with the base client from s01.

```csharp
IChatClient innerClient = new ChatClient(modelId,
    new ApiKeyCredential(apiKey),
    new OpenAIClientOptions { Endpoint = new Uri(baseUrl) })
    .AsIChatClient();
```

**Step 2**: Build the middleware pipeline.

```csharp
var client = innerClient
    .AsBuilder()
    .Use(inner => new TimingChatClient(inner))
    .Use(inner => new LoggingChatClient(inner))
    .Build();
```

`.AsBuilder()` creates a `ChatClientBuilder`. Each `.Use()` wraps the current client with a new middleware layer. The last `.Use()` added is the outermost layer — it sees requests first and responses last.

**Step 3**: Implement a middleware — TimingChatClient.

```csharp
class TimingChatClient(IChatClient innerClient)
    : DelegatingChatClient(innerClient)
{
    public override async Task<ChatResponse> GetResponseAsync(
        IEnumerable<ChatMessage> messages,
        ChatOptions? options = null,
        CancellationToken cancellationToken = default)
    {
        var sw = Stopwatch.StartNew();
        var response = await base.GetResponseAsync(
            messages, options, cancellationToken);
        sw.Stop();
        Console.WriteLine($"  [timing] {sw.ElapsedMilliseconds}ms");
        return response;
    }
}
```

The pattern is always the same:
1. Do something **before** calling `base.*`
2. Call `base.*` to pass through to the next layer
3. Do something **after** the response comes back
4. Return the response (or yield the streaming updates)

**Step 4**: Implement a middleware — LoggingChatClient.

```csharp
class LoggingChatClient(IChatClient innerClient)
    : DelegatingChatClient(innerClient)
{
    public override async Task<ChatResponse> GetResponseAsync(
        IEnumerable<ChatMessage> messages,
        ChatOptions? options = null,
        CancellationToken cancellationToken = default)
    {
        var lastUserMsg = messages
            .LastOrDefault(m => m.Role == ChatRole.User)?.Text;
        Console.WriteLine($"  [logging] Q: {lastUserMsg?[..50]}...");
        var response = await base.GetResponseAsync(
            messages, options, cancellationToken);
        Console.WriteLine($"  [logging] A: {response.Text?.Length} chars");
        return response;
    }
}
```

Each middleware is independent. You can test `LoggingChatClient` in isolation by wrapping a mock client. You can reuse it in any pipeline.

**Step 5**: Call the pipeline — the client works identically.

```csharp
var response = await client.GetResponseAsync(
    "What is dependency injection? One sentence.");
```

The caller doesn't know or care that middleware is present. The response is the same. The side effects (logging, timing) happen transparently.

---

## Pipeline Order Matters

```text
.Use(inner => new TimingChatClient(inner))    // outermost
.Use(inner => new LoggingChatClient(inner))   // middle
// OpenAI Client                              // innermost
```

**Request flow**: Timing → Logging → OpenAI
**Response flow**: OpenAI → Logging → Timing

Put cross-cutting concerns (logging, tracing, metrics) on the outside. Put business-logic middleware (retry, caching, function invocation) closer to the client.

MEAI provides built-in middleware you can insert anywhere in the pipeline:

| Middleware | What it does |
|-----------|-------------|
| `.UseFunctionInvocation()` | Automatically dispatches tool calls (see s04) |
| `.UseOpenTelemetry()` | Adds distributed tracing spans |
| `.UseDistributedCache()` | Adds response caching |

## What's Next

We have a provider-agnostic client with composable middleware. But the model can only answer questions — it cannot take actions in the real world.

→ s03 Agent Loop: The `while True` loop that lets the model call tools, execute them, and keep reasoning based on results.

<details>
<summary>Dive into Middleware Design</summary>

> The middleware pattern in MEAI follows the same decorator chain as ASP.NET Core middleware. Each `DelegatingChatClient` holds a reference to the `InnerClient` and delegates to it.

<details>
<summary>1. Why DelegatingChatClient, not events or hooks?</summary>

Events and hooks are decoupled but hard to reason about execution order. The decorator chain is explicit: you can see the exact order of middleware by reading the pipeline construction. Each layer wraps the next, making it trivial to test and compose.

</details>

<details>
<summary>2. Streaming requires separate override</summary>

`GetResponseAsync` handles non-streaming. `GetStreamingResponseAsync` handles streaming. You must override both if your middleware needs to observe all traffic. The streaming override uses `IAsyncEnumerable` and `yield return` to pass chunks through transparently.

```csharp
public override async IAsyncEnumerable<ChatResponseUpdate>
    GetStreamingResponseAsync(
        IEnumerable<ChatMessage> messages,
        ChatOptions? options = null,
        [EnumeratorCancellation] CancellationToken ct = default)
{
    // before
    await foreach (var update in base.GetStreamingResponseAsync(
        messages, options, ct))
    {
        yield return update;  // pass through
    }
    // after
}
```

</details>

<details>
<summary>3. FunctionInvokingChatClient is just middleware</summary>

Tool dispatch is not special. `FunctionInvokingChatClient` is a `DelegatingChatClient` that intercepts `tool_use` blocks, executes the tool, and feeds the result back into the conversation. It composes with logging and caching exactly like any other middleware. This is the subject of s04.

</details>

**In one sentence**: Middleware is the onion pattern applied to LLM calls — each layer adds behavior without changing the core.

</details>
